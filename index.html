QUIC protocol for resource-constraint devices

MABILLE Florent - August 2019

Hello everyone and welcome to this master thesis defence. Today I'll talk about the QUIC protocol for resource-constraint devices.

Motivations of QUIC for resource-constraint devices

  • Handshake delay
  • QUIC work in the user space
  • Reliability
  • Streams multiplexing
QUIC newly created protocol to reduce the latency of the web
delay -> connect two host easy
user space -> modifications
Reliability -> Every packet will be received by a retransmission mechanism
Stream multiplexing -> HOL problem

Resource-constraint devices

  • Small device with limited CPU, memory and battery
  • Different applications
    • Collect data
    • Accomplish an automated task
  • Challenges
    • Memory
    • Power consumption
    • Communication
    • Security
To complete the different possible tasks, there are different challenges.
Memory -> use the less memory we can.
Power consumption -> devices can work on a battery. The objective is to reduce the power used by the device and achieve good performance.
Communication -> devices have not enough power to reach the server and send data in an efficient way.
Security, devices do not have a lot of resources. As security uses lots of resources, we cannot do sophisticated protection on this kind of device.

Raspberry Pi 3b+ (CPU clock of 1.4 GHz and a RAM memory of 1GB)

Experiments

Methodology

  • Raspberry Pi as the client and server
  • Run with a Ethernet and a WiFi connection
  • Two implementations of QUIC
    • Quic-go
    • Picoquic
  • One HTTP/2 with TLS 1.3 in Go to compare (named https-go)
  • Different transfers of file (500 KB, 2 MB and 5 MB)
  • Add some load to simulate task on the Raspberry Pi
